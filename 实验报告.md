# C-- 编译器前端实验报告

## 1 项目概述

本项目实现了一个 C-- 语言的编译器前端，包含以下三个核心模块：

1. **词法分析器**：基于正则表达式 → NFA → DFA → 最小化 DFA 的经典流程，从源文件中识别 token 序列。
2. **SLR 语法分析器**：手写实现 SLR(1) 分析算法，从词法 token 序列构建抽象语法树（AST）。
3. **中间代码生成**：通过 C++ visitor 模式遍历 AST，调用自定义的中端 IR 库生成 LLVM IR（`.ll` 格式）。

整体编译流水线如下：

```
C-- 源码 (.sy) → 词法分析 → Token 流 → SLR 语法分析 → AST → C++ visitor → LLVM IR (.ll)
```

项目完全手写实现，未使用 Lex/Yacc/ANTLR 等自动代码生成工具，符合编译原理大作业要求。

## 2 词法分析器设计

### 2.1 正则定义

C-- 语言的词法规则由以下正则表达式定义（核心部分）：

| Token 类型 | 正则表达式 |
|-----------|-----------|
| 关键字 `int` | `int` |
| 关键字 `void` | `void` |
| 关键字 `return` | `return` |
| 关键字 `const` | `const` |
| 关键字 `float` | `float` |
| 关键字 `if` | `if` |
| 关键字 `else` | `else` |
| 标识符 | `[a-zA-Z_][a-zA-Z0-9_]*` |
| 整数字面量 | `[0-9]+` |
| 浮点数字面量 | `[0-9]+\.[0-9]+` |
| 双字符运算符 | `==` `<=` `>=` `!=` `&&` `||` |
| 单字符运算符 | `+` `-` `*` `/` `%` `=` `>` `<` |
| 界符 | `(` `)` `{` `}` `;` `,` |

关键字和标识符的优先级处理：当最长匹配到标识符模式时，需检查匹配内容是否为保留关键字，若是则返回对应 KW 编号，否则返回 IDN。

### 2.2 NFA 构造

NFA 采用 Thompson 构造法，支持以下节点类型：

- **CharNode**：单字符匹配
- **ConcatNode**：拼接（`ab`）
- **UnionNode**：选择（`a|b`）
- **ClosureNode**：克林闭包（`a*`）
- **CharSetNode**：字符集（`[a-z]`）

每个 NFA 状态包含出边映射（支持 ε 转移）和是否为终态的标记。对于多个正则表达式，通过创建超级初态（ε 连接到各 NFA 初态）和虚拟终态进行合并。

关键代码：`lexer/nfa.py` 中的 `NFA.build()` 和 `NFA.build_for_tokens()`。

### 2.3 DFA 确定化

采用子集构造法（Subset Construction）将 NFA 转换为 DFA：

1. 从 NFA 初态的 ε-closure 开始作为 DFA 初态
2. 对每个 DFA 状态和字母表中的每个字符，计算 `ε-closure(move(T, c))` 得到新状态
3. 使用 BFS 遍历直到不再产生新状态

对于 NFA 终态集合，DFA 状态需标记为终态，并选择优先级最高的 token 类型（关键字 > 运算符 > 标识符 > 数字）。

关键代码：`lexer/converter.py` 中的 `nfa_to_dfa()`。

### 2.4 DFA 最小化

采用 Hopcroft 分区算法对 DFA 进行最小化：

1. **初始划分**：将所有终态按 token 类型分为不同的组（不同 token 类型的终态不能等价），所有非终态归为一组
2. **迭代细分**：对每个分组，检查组内状态对每个输入字符的转移目标是否在同一分组。若不在，则按转移目标分组进行分割
3. **构建最小 DFA**：每个最终分组对应最小 DFA 的一个状态

最小化后的 DFA 状态数显著减少，提高了词法分析的运行效率。

关键代码：`lexer/converter.py` 中的 `minimize_dfa()` 和 `_split_partition()`。

### 2.5 最长匹配策略

词法分析时采用**最长匹配（Longest Match）**策略：从当前位置开始，沿 DFA 尽可能多地消费字符。每当经过一个终态时，记录当前匹配位置和 token 类型。当 DFA 无法继续转移时，回退到最后一个终态记录的位置，输出对应的 token。

例如输入 `inta`：DFA 先匹配到 `int`（终态），继续走到 `inta`（标识符终态）。由于 `inta` 更长，最终识别为标识符而非关键字。

### 2.6 关键字识别

关键字并非通过独立的正则模式优先识别，而是采用**后查表策略**：词法分析时先按标识符模式 `[a-zA-Z_][a-zA-Z0-9_]*` 进行最长匹配，匹配完成后检查匹配到的字符串是否在 `KEYWORDS` 字典中。若在，则将该 token 的类型从 `IDN` 改为 `KW`，并赋予对应编号；否则保留为 `IDN`。

例如输入 `int`：DFA 最长匹配到 `int`，终态类型为 `IDN`；查 `KEYWORDS` 表发现 `int` 对应编号 1，因此输出 `<KW,1>`。输入 `inta`：最长匹配到 `inta`，查表未命中，输出 `<IDN,inta>`。

### 2.7 输出格式

词法分析器的输出严格遵守大作业规范：

```
[单词符号]    <[种别],[内容]>
```

种别编号：

| 类别 | 编号 |
|------|------|
| KW: int, void, return, const, main, float, if, else | 1-8 |
| OP: + - * / % = > < == <= >= != && \|\| | 6-19 |
| SE: ( ) { } ; , | 20-25 |
| INT | 字面量值 |
| IDN | 标识符内容 |
| FLOAT | 字面量值 |

示例输出（`test_lexer.sy` 中 `int a = 10;`）：

```
int    <KW,1>
a      <IDN,a>
=      <OP,11>
10     <INT,10>
;      <SE,24>
```

## 3 语法分析器设计

### 3.1 EBNF 到 BNF 的改写

大作业附录给出了 C-- 语言的 36 条 EBNF 风格规则，包含可选符 `[]`、闭包 `{}` 等。SLR 分析器需要纯 BNF 文法，因此进行了以下改写：

| EBNF 构造 | BNF 改写 |
|----------|---------|
| 可选 `α?` | 拆分为两条产生式：`A → α` 和 `A → ε` |
| 重复 `{α}` | 改写为右递归：`A → αA | ε` |
| 表达式优先级 | 拆分为 `primaryExp → unaryExp → mulExp → addExp → relExp → eqExp → lAndExp → lOrExp → exp` |
| 左递归消除 | 采用 Tail 形式：`addExp → mulExp addExpTail`，`addExpTail → '+' mulExp addExpTail | ε` |

此外，将 `bType`（变量类型）和 `funcType`（函数类型）合并为统一的 `type`，并在 `topRest` 中区分函数定义和变量声明。

改写后的文法共 **98 条产生式**（含增广规则），**165 个 LR(0) 状态**，32 个终结符，54 个非终结符。

### 3.2 LR(0) 项目集规范族

LR(0) 项目形如 `A → α·β`，其中 `·` 表示分析进度。

**闭包计算（Closure）**：
- 若项目 `A → α·Bβ` 中 `B` 是非终结符，则将 `B` 的所有产生式 `B → ·γ` 加入闭包
- 迭代直到不再有新项目加入

**GOTO 函数**：
- `GOTO(I, X) = Closure({A → αX·β | A → α·Xβ ∈ I})`
- 即把所有 `·` 后面是 `X` 的项目，将 `·` 后移一位，再求闭包

**项目集族构建**：
1. 从增广文法的初始项目 `S' → ·Program` 的闭包开始
2. 对每个状态和每个文法符号 X，计算 GOTO(I, X)
3. 若得到新项目集，则作为新状态加入工作列表
4. 使用 BFS 直到工作列表为空

### 3.3 FIRST 和 FOLLOW 集

**FIRST 集**：
- 终结符的 FIRST 集是其自身
- 对于 `A → X₁X₂...Xₙ`，FIRST(A) 包含 FIRST(X₁) 中非 ε 的元素
- 若 X₁ 可推导出 ε，则继续考虑 FIRST(X₂)，依此类推
- 迭代计算直到收敛

**FOLLOW 集**（SLR 分析的关键）：
- 对于 `S' → Program`，将 `$` 加入 FOLLOW(Program)
- 对于产生式 `A → αBβ`，将 FIRST(β) 中非 ε 的符号加入 FOLLOW(B)
- 若 `A → αB` 或 β 可推导出 ε，则将 FOLLOW(A) 加入 FOLLOW(B)
- 迭代计算直到收敛

SLR 分析时，规约项目 `A → α·` 只对 FOLLOW(A) 中的终结符执行规约，从而解决部分冲突。

### 3.4 SLR 分析表构建

分析表分为 **ACTION 表**（终结符）和 **GOTO 表**（非终结符）：

- **移进（Shift）**：项目 `A → α·aβ`（a 为终结符），ACTION[state, a] = shift to GOTO(state, a)
- **规约（Reduce）**：项目 `A → α·`（规约项目），对 FOLLOW(A) 中每个终结符 t，ACTION[state, t] = reduce A → α
- **接受（Accept）**：项目 `S' → Program·`，ACTION[state, $] = accept

**冲突解决**：采用 **移进优先** 策略。先处理所有移进项目填入 ACTION 表，再处理规约项目，只在没有已有动作的位置填入 reduce。这自然解决了悬挂 else 等二义性问题。

### 3.5 SLR 分析流程

分析器维护三个同步栈：状态栈、符号栈和 AST 栈。

```
初始化: state_stack = [0], symbol_stack = ['$'], ast_stack = [None]
输入: Token 序列，末尾追加 $

循环:
  s = state_stack.top()
  a = 当前输入 token 对应的文法符号
  action = ACTION[s, a]

  case shift t:
    状态栈.push(t), 符号栈.push(a)
    AST 栈.push(叶子节点)
    输入指针++

  case reduce n (A → X₁X₂...Xₖ):
    从各栈弹出 k 个元素
    AST 栈.push(ASTNode(A, children=[弹出的 k 个子节点]))
    s' = state_stack.top()
    state_stack.push(GOTO[s', A])
    symbol_stack.push(A)

  case accept:
    返回 AST 栈顶节点（根节点）

  case error:
    报错并终止
```

每一步输出格式为：`[序号]    [栈顶符号]#[面临输入符号]    [执行动作]`

规约动作附带产生式编号，如 `reduction 77`。

## 4 AST 构建

### 4.1 构建方式

AST 节点在 SLR 规约过程中同步构建。分析器维护一个与状态栈、符号栈同步的 `ast_stack`：

- **移进时**：为终结符创建叶子节点（`Ident`、`IntConst` 等携带 `value`）
- **规约时**：弹出产生式右部对应数量的 AST 子节点，以产生式左部为节点名、子节点列表为 `children` 创建内部节点
- **接受时**：AST 栈顶节点即为完整语法树的根节点

AST 节点定义：

```python
class ASTNode:
    name: str          # 产生式左部（非终结符名）
    children: list     # 子节点列表
    value: str = ''    # 叶子节点的值（仅 Ident/IntConst 等）
```

### 4.2 AST 导出

分析器支持将 AST 导出为 JSON 格式，供 C++ visitor 使用：

```bash
python slr_parser/slr_parser.py tests/test_full.sy --ast-out tests/test_full.ast.json
```

JSON 结构示例（简化）：

```json
{
  "id": 0,
  "name": "Program",
  "children": [
    {
      "id": 1,
      "name": "compUnit",
      "children": [
        {
          "id": 2,
          "name": "topUnit",
          "children": [
            {"id": 3, "name": "type", "children": [{"id": 4, "name": "int"}]},
            {"id": 5, "name": "Ident", "value": "a"},
            ...
          ]
        },
        ...
      ]
    }
  ]
}
```

## 5 中间代码生成

### 5.1 Python 原型

初期使用 Python 在 `slr_parser.py` 中实现了一个 IR 生成器（`IRGenerator` 类），直接遍历 AST 生成 LLVM IR 文本。该原型验证了 IR 生成的基本逻辑，包括：

- 全局/局部变量声明（`alloca` / `store`）
- 函数定义与参数处理
- 表达式求值（通过 Tail 节点处理二元运算）
- if/else 条件分支
- return 语句

### 5.2 C++ visitor 接入中端

在 Python 原型验证通过后，将 AST 遍历和 IR 生成迁移到 C++ 端，接入提供的 `compiler_ir` 中端库。

**架构**：

```
AST JSON → ast_json.cpp (解析) → ir_visitor.cpp (遍历) → compiler_ir API → LLVM IR
```

**AST JSON 解析**（`ir_gen/ast_json.cpp`）：
- 手写轻量级 JSON 解析器
- 解析 AST JSON 文件，构建 `AstNode` 树（`name`、`value`、`children`）

**C++ visitor**（`ir_gen/ir_visitor.cpp`）：
- `CminusIRVisitor` 类包含 `visit_comp_unit`、`visit_top_unit`、`visit_stmt`、`visit_exp` 等方法
- 与 Python 原型一致，通过 Tail 节点处理表达式的二元运算
- 维护作用域栈（`scopes_`），支持局部变量和全局变量的区分
- 调用 `compiler_ir` 的 API 生成 IR：

| compiler_ir API | 用途 |
|----------------|------|
| `Module` | IR 模块容器 |
| `Function::create()` | 创建函数 |
| `BasicBlock::create()` | 创建基本块 |
| `IRBuilder::create_alloca()` | 分配局部变量 |
| `IRBuilder::create_store()` | 存储值 |
| `IRBuilder::create_load()` | 加载值 |
| `IRBuilder::create_iadd/isub/imul/isdiv()` | 二元算术运算 |
| `IRBuilder::create_icmp_*()` | 比较运算 |
| `IRBuilder::create_call()` | 函数调用 |
| `IRBuilder::create_cond_br()` | 条件分支 |
| `IRBuilder::create_br()` | 无条件分支 |
| `IRBuilder::create_ret()` | 返回值 |
| `GlobalVariable::create()` | 全局变量 |

**语义检查**：
- 重复定义检测：同一作用域内不允许同名变量
- 未定义变量检测：使用前必须声明
- const 赋值保护：const 变量不可修改

**编译与运行**：

```bash
cd final_project/ir_gen/build
cmake ..
make

./ir_gen_demo ../tests/test_full.ast.json --out output.ll
./ir_gen_demo --demo    # 运行内置示例
```

### 6.4 test_return_if.sy

**源码**：
```c
int main() {
    int x = 1;
    if (x > 0) {
        return x;
    } else {
        return 0;
    }
    x = 2;
    return x;
}
```

**LLVM IR 输出**（验证 return 后无多余 br）：
```llvm
; ModuleID = 'sysy2022_compiler'
source_filename = "test_return_if.sy"

define i32 @main() {
label_1:
  %op0 = alloca i32
  store i32 1, i32* %op0
  %op1 = load i32, i32* %op0
  %op2 = icmp sgt i32 %op1, 0
  br i1 %op2, label %label_2, label %label_3

label_2:
  %op3 = load i32, i32* %op0
  ret i32 %op3

label_3:
  ret i32 0
}

if/else 两个分支均以 `ret` 终结，后续 `x = 2; return x;` 因 `is_terminated()` 检测已跳过，不会产生 unreachable 的 `br` 指令或死代码，IR 结构合法。

## 附录 A：SLR 文法产生式表

以下为 `print_grammar_report.py` 输出的完整产生式表：

```
SLR 文法产生式表
================================================================================
0       S' -> Program
1       Program -> compUnit
2       compUnit -> topUnit compUnit
3       compUnit -> ε
4       topUnit -> constDecl
5       topUnit -> type Ident topRest
6       topRest -> ( funcFParamsOpt ) block
