# 编译原理大作业小组分工

## 分工原则

本项目按照编译器前端的处理流程进行分工：词法分析、SLR 语法分析、AST 与流水线集成、中间代码生成。各成员负责的模块边界清晰，便于开发、测试和展示。

## 成员分工

### 成员 A：词法分析器

负责内容：

- 设计 C-- 语言的 token 正则定义，包括关键字、标识符、整数、浮点数、运算符和界符。
- 实现正则表达式到 NFA 的构造。
- 实现 NFA 到 DFA 的确定化。
- 实现 DFA 最小化。
- 实现最长匹配、关键字识别和 token 输出格式。
- 编写词法分析测试样例。
- 完成报告中的词法分析器设计部分。

对应目录：

```text
final_project/lexer/
```

建议工作量占比：20%。

### 成员 B：SLR 语法分析器

负责内容：

- 将 PDF 附录中的 EBNF 文法改写为适合 SLR 分析的 BNF 文法。
- 实现 FIRST 集和 FOLLOW 集计算。
- 实现 LR(0) 项目集规范族构造。
- 实现 SLR ACTION/GOTO 分析表构造。
- 实现移进、规约、接受和错误处理逻辑。
- 输出带产生式编号的规约过程。
- 编写文法报告工具，输出产生式编号表、FIRST/FOLLOW 集和文法改写说明。
- 完成报告中的语法分析器设计部分。

对应文件：

```text
final_project/slr_parser/slr_grammar.py
final_project/slr_parser/slr_items.py
final_project/slr_parser/slr_table.py
final_project/slr_parser/print_grammar_report.py
```

建议工作量占比：30%。

### 成员 C：AST 构建、流水线集成与测试

负责内容：

- 在 SLR 规约过程中构建 AST。
- 实现 AST JSON 导出功能。
- 集成词法分析、语法分析、AST 构建和 IR 输出流程。
- 维护命令行参数，例如 `--ir-out` 和 `--ast-out`。
- 组织测试用例，覆盖基本函数、变量、常量、if/else、return 等场景。
- 编写项目 README 和运行说明。
- 汇总测试结果，完成测试报告。

对应文件和目录：

```text
final_project/slr_parser/slr_parser.py
final_project/tests/
final_project/README.md
```

建议工作量占比：20%。

### 成员 D：C++ Visitor 与中间代码生成

负责内容：

- 设计 C++ 中间代码生成模块。
- 实现 AST JSON 解析器。
- 实现 C++ visitor，访问 AST 节点并调用 `compiler_ir` 中端接口。
- 调用 `Module`、`Function`、`BasicBlock`、`IRBuilder`、`GlobalVariable` 等类生成 LLVM IR。
- 实现变量、常量、函数、参数、函数调用、赋值、算术表达式、比较表达式、if/else 和 return 的 IR 生成。
- 增加基础语义检查，包括重复定义、未定义变量和给 const 赋值。
- 在 WSL 环境下编译并验证 C++ visitor。
- 完成报告中的中间代码生成部分。

对应目录：

```text
final_project/ir_gen/
compiler_ir/
```

建议工作量占比：30%。

## 展示分工建议

PPT 展示可按照编译流程进行：

```text
成员 A：源代码 -> token，介绍词法分析器和自动机构造。
成员 B：token -> SLR 规约过程，介绍文法改写、LR(0) 项目集和 SLR 表。
成员 C：规约过程 -> AST，介绍 AST 构建、AST JSON、测试样例和整体运行流程。
成员 D：AST -> LLVM IR，介绍 C++ visitor 和 compiler_ir 中端调用。
```

## 工作量汇总

```text
成员 A：词法分析器与词法测试，20%
成员 B：SLR 语法分析器与文法处理，30%
成员 C：AST、流水线集成与测试报告，20%
成员 D：C++ visitor 与 LLVM IR 生成，30%
```

该分配方式兼顾模块独立性和实现难度，能够体现每位成员在项目中的明确贡献。
